## Context

当前 GoEdge + CoreDNS 解析链路已具备基础 DNS 调度能力，但请求来源主要使用递归解析器地址，无法可靠代表最终用户位置。上游已支持通过 EDNS Client Subnet (ECS) 透传客户端网络信息，因此本次变更需要跨 CoreDNS 请求解析、GoEdge 调度和 IP 库映射三个模块实现“按 ECS 原始 IP 做地市级路由”。

约束条件：
- 必须兼容无 ECS 请求，不能影响现有解析可用性。
- 必须兼容 IPv4 与 IPv6 ECS 前缀，且对异常/伪造 ECS 有安全降级。
- 必须尽量复用 GoEdge 现有 IP 库与区域数据（country/province/city），避免引入额外地理库。

## Goals / Non-Goals

**Goals:**
- 从 DNS 请求中稳定提取 ECS 原始客户端 IP，并在请求上下文中透传。
- 将客户端 IP 映射到地市（cityId/cityName）并用于节点调度优选。
- 当城市无可用节点或 ECS 不可用时，按既有策略自动回退，保证可用性。
- 提供可观测性（命中、降级、异常）以支撑灰度与回归验证。

**Non-Goals:**
- 不在本变更中重构全部 DNS 调度算法，只增加 ECS 城市优选分支。
- 不新增外部第三方 GeoIP 服务依赖。
- 不改变上游 DNS 厂商 ECS 透传协议语义。

## Decisions

1. **在 CoreDNS 入站请求阶段解析 ECS，并只选择一个“有效客户端 IP”进入上下文**
- Decision: 优先读取 ECS 地址；若缺失或非法则降级为当前请求来源 IP。
- Rationale: 将“来源解析”前置并标准化，可降低后续调度逻辑复杂度。
- Alternative considered:
  - 在调度阶段再解析 ECS：会造成多处重复解析与不一致处理。

2. **城市映射复用 GoEdge 现有 IP 库数据结构**
- Decision: 复用已有 region/city/provider 体系完成 IP -> cityId 映射。
- Rationale: 避免多份地域数据导致冲突，并降低维护成本。
- Alternative considered:
  - 引入独立 GeoIP 库：更新链路更复杂，且与现有区域模型难对齐。

3. **节点选择采用“城市优先 + 逐级回退”策略**
- Decision: 先筛选城市匹配且健康的边缘节点；若为空，回退到省级/大区或现有默认策略。
- Rationale: 在提升就近性的同时保证无中断。
- Alternative considered:
  - 强制城市命中：在城市节点不足时会直接降低可用性。

4. **通过显式开关控制 ECS 城市路由启用范围**
- Decision: 在 DNS 提供商配置中增加开关与（可选）回退策略参数。
- Rationale: 支持灰度发布、回滚和按业务域名分批启用。
- Alternative considered:
  - 全局默认启用：变更风险高，不利于逐步验证。

## Risks / Trade-offs

- [ECS 缺失或被中间链路剥离] → 记录降级原因并自动使用现有路由策略，保证解析结果可用。
- [IP 库数据过旧导致城市映射偏差] → 建立数据更新时间监控与回归样例（重点城市 IP 样本）。
- [城市级过滤导致候选池变小] → 采用健康检查与分级回退，必要时可配置直接跳过城市过滤。
- [客户端隐私与合规风险] → 仅在调度链路使用 ECS IP，不落长期明文日志（或脱敏存储）。

## Migration Plan

1. 增加配置项并默认关闭 ECS 城市路由。
2. 在测试环境启用，验证典型样本（如 `61.142.56.193 -> 广州`）命中与降级路径。
3. 小流量灰度到指定域名/线路，观察命中率、解析耗时和错误率。
4. 全量启用后持续监控，若异常升高可一键关闭开关回滚到原路由。

## Open Questions

- 城市无节点时的第一回退层级是“省级优先”还是“全局权重优先”？
- IPv6 ECS 前缀（非 `/128`）是否按前缀首地址映射，还是仅在精确匹配时启用城市路由？
- 是否需要在管理端暴露“按城市命中统计”用于运营可视化？
